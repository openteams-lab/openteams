### 分篇拆解与交付规划

### 1. 产品价值向

- 建议标题：OpenTeams 是什么：为什么 AI 时代需要一个真正的多 Agent 团队工作台
- 目标读者：对 AI Coding Agent 感兴趣，但仍停留在单 Agent 或多窗口切换模式的开发者与团队负责人
- 文章定位：面向冷启动用户的认知建立文章，核心是把 OpenTeams 从“又一个 AI 工具”讲成“多 Agent 协作基础设施”
- 建议文件名：`01-what-is-openteams-and-why-multi-agent-matters.md`
- 核心论点：
  1. 单 Agent 能回答问题，但难以模拟真实团队分工。
  2. 多窗口工作流可以拼凑协作，但上下文割裂、切换成本高。
  3. OpenTeams 的价值在于共享群聊上下文、并行执行、跨模型协作与本地运行。
- 可直接引用的仓库能力：支持 10 种 coding-agent runtime、共享群聊上下文、并行执行、Agent `@mention` 协作、完全本地执行、桌面端 + npx Web 入口
- 推荐结构：问题背景 -> 现有方案痛点 -> OpenTeams 的差异 -> 适合谁用 -> 快速开始入口
- 交付建议：
  - 篇幅：1800-2200 字
  - 配图：产品总览图 1 张，对比表 1 张，群聊协作动图 1 处
  - CTA：引导读者先用 `npx openteams-web` 或桌面版体验第一支团队

### 2. 协作场景向

- 建议标题：从单人提问到团队协作：用 OpenTeams 跑通一次真实的多 Agent 交付流程
- 目标读者：希望把 AI 用进日常交付流程的产品经理、创业团队、内容与研发混编团队
- 文章定位：用一个具体任务展示产品不是“多人聊天”，而是“多角色协作执行”
- 建议文件名：`02-real-multi-agent-collaboration-workflow.md`
- 推荐场景：让工程 Agent、研究 Agent、写作 Agent、营销 Agent 在同一会话里完成一个功能发布或内容交付
- 可直接引用的仓库能力：160+ 内置 AI 成员、8 个团队预设、团队规则配置、会话归档/恢复、工作项、附件、运行日志、Diff 查看
- 核心论点：
  1. 共享上下文减少重复解释。
  2. `@mention` 让交接显式化。
  3. 团队规则让协作从自由聊天升级为可控流程。
  4. 运行记录和 Diff 让结果可追踪。
- 推荐结构：任务设定 -> 组建团队 -> 分工执行 -> 中途交接 -> 查看产出 -> 复盘收益
- 交付建议：
  - 篇幅：2000-2600 字
  - 配图：会话页截图、成员侧栏、Diff/Run History 截图
  - CTA：引导用户从“成员预设 + 团队预设”开始复用此流程

### 3. 技术亮点向

- 建议标题：OpenTeams 的底层是怎么搭起来的：Rust 后端、React 前端与本地运行时如何协同
- 目标读者：技术决策者、开源开发者、潜在贡献者
- 文章定位：架构可信度文章，回答“为什么它能支撑多 Agent 会话、实时流式更新和本地工作空间隔离”
- 建议文件名：`03-openteams-architecture-rust-react-local-runtime.md`
- 可直接引用的仓库能力：Rust workspace、React + TypeScript + Vite 前端、Tauri 桌面端、WebSocket 实时流、SQLx 数据模型、workspace 级 `.openteams/` 运行时目录、共享类型生成
- 可展开的技术点：
  1. chat session / agent / message / run / permission / skill / work item 等核心模型如何支撑协作。
  2. 前后端通过生成的 shared types 对齐数据结构。
  3. 本地工作空间与运行记录路径如何帮助隐私与可审计性。
  4. WebSocket 流式事件如何支撑实时状态更新。
- 推荐结构：总体架构 -> 数据模型 -> 运行时目录 -> 实时事件 -> 为什么选择这套技术栈
- 交付建议：
  - 篇幅：2200-3000 字
  - 配图：系统架构图 1 张，数据流图 1 张
  - CTA：引导开发者阅读 `README.md`、`AGENTS.md` 与相关 `crates/` 目录

### 4. 团队效率向

- 建议标题：不再手动复制粘贴：OpenTeams 如何把多模型协作变成可管理的团队效率系统
- 目标读者：已重度使用 AI 编程工具、正在管理多人研发协作的技术团队
- 文章定位：强调效率收益与组织可管理性，适合投放在开发者社区、团队管理者社群
- 建议文件名：`04-how-openteams-improves-team-efficiency.md`
- 可直接引用的仓库能力：多模型协作、共享上下文、并行执行、会话状态管理、运行历史、日志、Diff、工作项、会话归档/恢复、权限模型
- 核心论点：
  1. 效率不只来自更强模型，也来自更少的上下文搬运。
  2. 并行与分工让多个 Agent 同时推进不同子任务。
  3. 记录、日志、Diff、工作项让效率提升可见且可管理。
  4. 权限与工作空间隔离降低协作失控风险。
- 推荐结构：传统多 Agent 工作流成本 -> OpenTeams 的效率杠杆 -> 管理视角的收益 -> 适用团队阶段
- 交付建议：
  - 篇幅：1800-2400 字
  - 配图：效率前后对比图、运行记录截图
  - CTA：引导团队用一个真实需求试跑“并行协作 + 结果复盘”

### 5. 生态扩展向

- 建议标题：不只是一套界面：OpenTeams 如何用预设成员、技能库与自定义扩展长出生态
- 目标读者：希望深度定制 AI 团队的开发者、生态合作方、社区贡献者
- 文章定位：生态介绍与路线延展文章，强调平台化潜力，而不是只卖单个功能点
- 建议文件名：`05-openteams-members-skills-and-extension-ecosystem.md`
- 可直接引用的仓库能力：160+ 内置 AI 成员、8 个团队预设、1000+ 内置技能、技能导入、允许为 session agent 配置 `allowed_skill_ids`、技能注册与管理服务
- 核心论点：
  1. 成员预设降低组队成本。
  2. 团队预设降低流程搭建成本。
  3. 技能库与技能导入让 Agent 从“会聊天”变成“可扩展执行单元”。
  4. 平台的长期价值在于角色、流程、能力三层一起扩展。
- 推荐结构：为什么需要生态层 -> 成员预设 -> 团队预设 -> 技能扩展 -> 社区与未来方向
- 交付建议：
  - 篇幅：1800-2300 字
  - 配图：成员预设界面、技能面板、团队导入流程图
  - CTA：引导读者从“先选团队，再选成员，再挂技能”理解产品扩展路径

### 建议发布顺序

1. 先发产品价值向，建立认知。
2. 第二篇发协作场景向，把抽象概念落到真实使用流程。
3. 第三篇发团队效率向，接住已经有 AI 使用经验的人群。
4. 第四篇发技术亮点向，建立开发者与贡献者信任。
5. 最后发生态扩展向，承接社区与长期价值叙事。

### 统一写作口径

- 避免写成“万能 AI 自动公司”，应强调当前已验证的是多 Agent 共享上下文、协作执行与本地工作流。
- 对能力描述尽量锚定仓库中已经明确存在的模块与页面，如 Chat Sessions、Workspaces、Project Kanban、Skills、Team Presets。
- 对尚在 roadmap 的内容要单独标注为“路线图”或“未来方向”，不要写成既有能力。
- 每篇都建议保留一个真实界面截图或仓库结构引用，增强可信度。
