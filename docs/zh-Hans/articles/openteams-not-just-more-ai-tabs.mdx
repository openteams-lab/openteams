## 导语

很多人第一次看到多 Agent 产品，第一反应都很直接：这不就是把几个 AI 工具同时开着用吗？

这个理解不能说错，但只说对了一半。因为当任务开始跨越研究、写作、编码、审查和交付时，真正拖慢团队的，通常已经不是模型本身，而是上下文怎么同步、任务怎么交接、过程怎么追踪。

我看完 openteams 的公开仓库后，最大的感受是：它想解决的不是“让你同时问更多模型”，而是“怎么把多个 Agent 真正组织成一个能协作的小团队”。这件事听起来像产品措辞差异，实际上对应的是两种完全不同的工作方式。

> ASCII 配图落位说明 1：放在导语之后，用来快速建立“多开窗口”和“团队协作”之间的结构差异。

```text
多开 AI 窗口

You
 |- Window A -> answer A
 |- Window B -> answer B
 `- Window C -> answer C

openteams 团队协作

You
  |
  v
Shared Session
  |- Researcher -> facts
  |- Writer     -> draft
  |- Reviewer   -> fixes
  `- Editor     -> final output
```

## 多开几个窗口，解决的是数量问题

如果你的需求只是“同一个问题多问几个模型，看看谁答得更好”，那多开窗口已经足够了。

- 你想快速比较不同模型的回答风格
- 你想让一个模型补充另一个模型的盲区
- 你处理的是一次性问题，而不是需要多人接力的任务

在这种场景下，核心变量确实只是模型数量。谁更强、谁更快、谁更便宜，往往比协作结构更重要。

所以很多人会自然得出一个判断：多 Agent 产品不过是把“多模型调用”重新包装了一遍。

这个判断的问题在于，它默认你的任务始终是单点式的、线性的、一次性完成的。

但只要任务进入稍微复杂一点的阶段，瓶颈很快就会从“谁来回答”变成“谁来协作”。

## 真正卡住团队的，往往不是模型，而是协作摩擦

真实工作里，最耗人的通常不是没人输出，而是下面这些环节：

1. 你要自己拆任务。
2. 你要自己把上一轮上下文复制给下一个模型。
3. 你要自己判断谁适合做研究、谁适合写作、谁适合改代码。
4. 你要自己记录到底谁做了什么，方便后面复盘和接力。

单 Agent 很强，但很多时候它更像一个能力很全面的个人贡献者。只要工作开始跨角色，真正上升的成本不是提问成本，而是**切换成本、搬运成本和返工成本**。

这也是为什么我会说，多 Agent 团队如果要成立，关键不是“多”，而是“能不能像团队一样工作”。

## openteams 更像是在解决协作结构问题

从仓库公开的信息看，openteams 至少抓住了多 Agent 产品里最容易被忽略、却最关键的四件事。

### 1. 共享上下文，而不是人工转述

README 里写得很明确：多个参与者在同一段共享会话历史里工作，而不是在不同窗口之间靠人工复制粘贴提示词。这个能力看起来基础，实际上决定了多 Agent 协作是不是能成立。

因为只要上下文还依赖人来搬运，所谓多 Agent 很容易退化成“多个标签页 + 一个很累的项目经理”。

### 2. 直接 `@mention`，而不是所有任务都靠人串联

openteams 允许成员在会话里互相 `@mention`。这意味着任务交接不再全部依赖用户自己转发，而是可以在同一条协作链路里完成。

这个机制的价值不在于“更像聊天软件”，而在于它降低了人类作为中控节点的负担。你不需要每一步都重复描述背景、指定下一位执行者、再重新解释上下文。

### 3. 并行执行，而不是顺序排队

仓库 README 提到，多名 Agent 可以在同一共享会话里并行处理任务。对内容生产、需求分析、代码评审、文档整理这类天然可拆分的工作来说，并行的意义不只是节省时间，更是改变了任务结构。

原来你必须按顺序等待：先研究、再写作、再校对、再整理。现在则有机会把这些环节拆成可同时推进的子任务。

### 4. 执行留痕，而不是结果出来就结束

openteams 的聊天系统不只是保留消息，还会记录运行相关的日志、diff、上下文文件和产物路径，运行时数据落在工作区里的 `.openteams/` 目录。

这件事非常关键。因为在团队协作里，结果重要，但过程同样重要：

- 谁查了哪些事实
- 谁改了哪些文件
- 哪一步失败了
- 后面的人能不能接着做

只有过程可见，协作才具备复盘和接力的基础。否则所谓团队，最后还是一次次彼此隔离的单轮问答。

> ASCII 配图落位说明 2：放在“执行留痕”段落之后，用来把可追踪资产具体化，方便读者理解它不是抽象概念。

```text
<workspace>/
`-- .openteams/

    `-- runs/<session_id>/run_records/
        `-- session_agent_<id>_run_0001/
            +-- input.md
            +-- output.md
            +-- raw.log
            `-- diff.patch
```

## 一个更接近真实工作的例子

假设你要为一个开源产品做一次完整内容发布，不是写一条宣传文案，而是要同时产出几篇文章，并且兼顾技术准确性、传播视角、案例表达和配图说明。

传统做法通常是这样的：

- 先找一个模型整理卖点
- 再找另一个模型改成技术稿
- 再找第三个模型改成传播稿
- 最后自己把不同版本拼起来

看起来也能完成，但这里最大的瓶颈往往不是模型不够聪明，而是**你始终是那条最窄的通道**。所有信息都要经过你转发，所有角色切换都要靠你重新组织。

而在 openteams 这种协作模式里，这件事可以被组织成一个小团队：有人做选题拆解，有人做事实核验，有人写技术稿，有人补传播稿，有人负责知乎化改写，还有人做 ASCII 配图方案。不同成员共用同一上下文，彼此能在同一会话里交接，最后再由编辑统一收口。

这时候你节省的，不只是写作时间，而是那部分最容易被忽略的时间：

- 重复解释背景的时间
- 在不同窗口来回复制内容的时间
- 因上下文断裂产生返工的时间
- 因过程不可见而额外校对的时间

## 所以判断多 Agent 产品，要看哪四个问题

我现在越来越倾向于用下面四个问题判断一个多 Agent 产品到底是在做演示，还是在做真正可用的协作系统：

1. 上下文是不是共享的，而不是靠人工转述。
2. 角色是不是可分工的，而不是一个 Agent 从头做到尾。
3. 任务是不是可交接的，而不是所有链路都必须经过人。
4. 过程是不是可追踪的，而不是最后只剩一段答案。

从这个标准看，openteams 的方向是清楚的。它不只是在做一个模型聚合界面，而是在把“团队协作”这件事产品化。仓库里也能看到这种路线：除了多 Agent 群聊、共享上下文和并行执行，还有 160+ 内置 AI 成员、8 个团队预设、1000+ 技能库，以及团队规则配置。

翻译得更直白一点就是：它不只关心“谁能做”，也关心“这些角色如何配合”。

## 当然，它也不是适合所有任务

说到这里，也需要把边界讲清楚。

多 Agent 协作并不天然适合所有工作。如果你的任务本身很短、很线性、几乎不需要分工，一个强单 Agent 通常更直接。比如改一个函数名、补一个小脚本、写一段 SQL，让一个模型一口气做完，往往更省心。

多 Agent 真正有价值的场景，通常同时满足三个条件：

- 任务可以拆开，并且拆开后能并行推进
- 不同环节确实需要不同能力，比如研究、编码、文案、审查
- 你除了要结果，还要过程可回看、可交接、可复盘

如果不满足这三点，团队化很可能只是看起来热闹，实际却增加管理成本。

## 回到那个最常见的问题

所以，openteams 到底是不是“多开几个 AI 窗口”？

我的答案是：表面上它当然也支持同时使用多个 Agent，但它真正想解决的，不是聊天窗口数量问题，而是**多角色协作的组织问题**。

这件事有没有价值，也不取决于你对“多模型”有没有新鲜感，而取决于你的工作是不是已经进入下一个阶段：一个人问一个模型已经不够，真正卡住你的是任务拆分、上下文同步、跨角色交接和执行留痕。

如果你还停留在单点提问，多开窗口也许已经够了。

但如果你已经开始遇到协作链路的问题，那 openteams 这类产品思路，至少值得认真看一眼。它未必替代单 Agent，却很可能定义了下一阶段更接近真实团队工作的 AI 使用方式。

而这，恰恰是一个比“模型又变强了”更值得讨论的话题。
