# openteams-cli 瘦身方案

**基准**: 当前二进制约 144MB

---

## 一、保守版方案

### 目标体积
**80-90MB** (减少约 40-55MB)

### 移除/优化项

| 序号 | 变更项 | 节省体积 | 实施方式 |
|------|--------|----------|----------|
| 1 | Bun 编译优化 | 5-10MB | 启用 minify, tree-shaking |
| 2 | models-snapshot 外置 | 1.5MB | 改为运行时加载 + 本地缓存 |
| 3 | shiki → highlight.js | 2-3MB | 替换代码高亮库 |
| 4 | 非默认 AI 提供商延迟加载 | 8-12MB | 仅内置 OpenAI/Anthropic，其他按需导入 |
| 5 | @parcel/watcher 多平台二进制裁剪 | 3-5MB | 仅打包目标平台原生模块 |

### 保留能力
- ✅ 完整 TUI 界面
- ✅ 所有 AI 提供商 (延迟加载)
- ✅ 本地服务器
- ✅ MCP 协议支持
- ✅ GitHub 集成
- ✅ Effect 框架

### 对核心 CLI 影响
| 影响项 | 程度 | 说明 |
|--------|------|------|
| 首次启动 | 轻微 | models 数据需网络获取或缓存 |
| 非默认提供商首次调用 | 轻微 | 需动态加载 (已有网络延迟，影响小) |
| 代码高亮效果 | 轻微 | highlight.js 略逊于 shiki，但可接受 |
| 功能完整性 | 无 | 所有功能保留 |

### 实施复杂度
**低** - 主要是构建配置和模块加载逻辑调整

---

## 二、激进版方案

### 目标体积
**50-60MB** (减少约 80-90MB)

### 移除/优化项

| 序号 | 变更项 | 节省体积 | 实施方式 |
|------|--------|----------|----------|
| 1 | Bun 编译优化 | 5-10MB | 启用 minify, tree-shaking |
| 2 | models-snapshot 外置 | 1.5MB | 改为运行时加载 |
| 3 | shiki 移除 | 2-3MB | 使用 Tree-sitter 高亮或无高亮 |
| 4 | 仅保留 2 个默认 AI 提供商 | 12-15MB | OpenAI + Anthropic，其他完全移除 |
| 5 | TUI 可选化 | 5-8MB | 提供 `--no-ui` 模式，headless 优先 |
| 6 | @parcel/watcher 裁剪 | 3-5MB | 仅目标平台 |
| 7 | Effect 框架移除 | 3-5MB | 重构为原生 async/Promise |
| 8 | 本地服务器可选 | 5-8MB | 拆分为 openteams-server |
| 9 | GitHub 集成可选 | 2-3MB | 拆分为 @openteams/github 插件 |
| 10 | hono-openapi 移除 | 1-2MB | 非必需依赖 |

### 保留能力 (核心 CLI)
- ✅ 基础命令行交互
- ✅ OpenAI/Anthropic 提供商
- ✅ 文件操作工具
- ✅ 代码解析 (Tree-sitter)
- ⚠️ TUI 需显式启用
- ⚠️ 本地服务器需独立安装
- ⚠️ 其他 AI 提供商需安装插件包

### 可选扩展包
```
@openteams/provider-openai      # 默认内置
@openteams/provider-anthropic   # 默认内置
@openteams/provider-google      # 可选安装
@openteams/provider-azure       # 可选安装
@openteams/provider-*           # 其他提供商
@openteams/tui                  # TUI 界面
@openteams/server               # 本地服务器
@openteams/github               # GitHub 集成
```

### 对核心 CLI 影响
| 影响项 | 程度 | 说明 |
|--------|------|------|
| 默认功能集 | 中等 | 仅核心功能，需安装扩展包 |
| 用户安装体验 | 中等 | 可能需要额外安装步骤 |
| TUI 启动 | 中等 | 需显式命令或参数 |
| 开发维护 | 高 | 需维护多个包 |

### 实施复杂度
**高** - 需要架构重构、包拆分、插件系统设计

## 三、对比总结

| 维度 | 保守版 | 激进版 |
|------|--------|--------|
| 目标体积 | 80-90MB | 50-60MB |
| 减少比例 | ~35% | ~60% |
| 功能完整性 | 100% | 核心 + 插件扩展 |
| 用户体验影响 | 低 | 中 |
| 实施周期 | 1-2 周 | 1-2 月 |
| 维护复杂度 | 低 | 高 |
| 可逆性 | 高 | 低 (需重构) |

## 四、建议优先落地方案

### 推荐: 保守版

**理由**:
1. **投入产出比高**: 1-2 周可获得 35% 体积减少
2. **风险可控**: 不影响功能完整性
3. **可渐进**: 可在保守版基础上逐步推进激进版
4. **用户友好**: 无需改变使用习惯

### 实施路径

```
Phase 1 (第1周):
├── 启用 Bun minify + tree-shaking
├── models-snapshot 改为运行时加载
└── 预期减少: ~10MB

Phase 2 (第2周):
├── shiki → highlight.js
├── AI 提供商延迟加载机制
└── 预期减少: 再 ~15MB

Phase 3 (后续优化):
├── @parcel/watcher 平台裁剪
├── 其他小型优化
└── 预期减少: 再 ~10MB
```

### 后续演进

保守版落地后，可基于用户反馈决定是否推进激进版:
- 若体积仍不满意 → 推进激进版 Phase 1 (TUI 可选化)
- 若扩展需求强烈 → 设计插件系统
- 若维护成本高 → 保持保守版

---

## 五、风险与缓解

### 保守版风险
| 风险 | 缓解措施 |
|------|----------|
| models 网络依赖 | 本地缓存 + 离线降级 |
| 动态加载延迟 | 预加载提示 + 后台预取 |
| 高亮效果变化 | 可配置高亮引擎 |

### 激进版风险
| 风险 | 缓解措施 |
|------|----------|
| 用户困惑 (功能缺失) | 安装向导 + 自动检测推荐 |
| 插件生态碎片化 | 官方维护核心插件 |
| 文档复杂度 | 分层文档 (核心/扩展) |

---

## 六、结论

**优先实施保守版**，预期 2 周内将体积从 144MB 降至 80-90MB，减少约 35-40%，同时保持功能完整性。

激进版作为长期目标，可在保守版验证后，根据用户反馈和业务需求逐步推进。
