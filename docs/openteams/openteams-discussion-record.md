# OpenTeams 讨论记录

## 文档目的

这份文档用于整理本轮关于 OpenTeams 的产品讨论，沉淀当前已经形成的产品定义、边界判断、治理机制和实现约束。

## 1. OpenTeams 正在变成什么

讨论已经明确偏离了“多 Agent 群聊工具”这个方向。

当前较稳定的定义是：

- OpenTeams 是一个以人为中心的多 Agent 协作系统。
- 群聊是人进入协作系统的统一入口，但不是产品本体。
- 产品本体是治理层，它负责把复杂协作压缩成可控、可审计、可收束的流程。
- 治理层之下是执行层，执行层可以运行独立的多 Agent 工作流。

一句话概括：

> OpenTeams 卖的不是更多对话，而是可验证的协作质量。

## 2. 核心产品命题

这个产品只有在同时做到两件事时才成立：

1. 提高信息密度。
2. 降低决策成本。

这意味着人不应该成为系统里的又一个执行节点，而应该扮演：

- 目标设定者
- 边界设定者
- 分歧仲裁者
- 最终责任人
- 结果验收者

人是导演、裁判和责任主体，而不是被消息淹没的群管理员。

## 3. 多 Agent 群协作真正有价值的场景

讨论中识别了 3 个最值得优先验证的工作：

1. 高风险决策评审。
2. 故障处理与复杂排错。
3. 内容生产与上线协同。

这 3 类工作的共同特征是：

- 多角色必须同时参与
- 分歧本身有价值
- 上下文必须共享
- 交接过程必须可见
- 最终仍然需要人拍板

## 4. 群聊不是核心，但仍然有意义

讨论最后形成的是一个更克制的判断：

- 群聊不是普适最优界面
- 群聊不应该被当成默认产品定义
- 群聊真正适用于高不确定、高耦合、高对齐成本的问题
- 群聊的价值在于让人可以随时打断、改方向、观察交接、保留协作现场感

因此正确关系是：

- 治理层 = 产品核心
- 群聊 = 治理层的解释界面和介入界面

## 5. 治理层的定义

治理层从执行和信息两个角度都被具体化了。

### 5.1 执行治理

治理层可拆成 4 个具体系统：

1. 准入器：什么问题可以进入协作模式。
2. 路由器：谁必须发言，谁不该发言，何时需要升级。
3. 决策账本：记录事实、共识、分歧、待拍板项和责任归属。
4. 退出器：何时结束讨论并切回执行或个人流程。

### 5.2 信息治理

噪声的根源不是“说得太多”，而是不同价值密度的消息被放进同一条流，并且视觉权重相同。

所以治理层必须控制：

- 消息分型
- 结构要求
- 摘要规则
- 升降级规则
- 消息向决策、任务、责任项的提炼机制

## 6. 四个治理问题

所有群聊功能都应该先回答四个问题：

1. 为什么这个问题要进入协作模式？
2. 谁负责收束讨论？
3. 什么时候退出协作并回到执行？
4. 谁对结果负责？

如果一个功能回答不了这四个问题，它就只是界面装饰，不是治理机制。

## 7. 工作模式与开放模式

用户提出了“双模式”结构，讨论最终接受了这个结构，但同时明确了价值层级。

### 开放模式

- 适合闲聊、娱乐、探索、灵感碰撞
- 可以容忍较多信息噪声
- 更像活跃和入口层

### 工作模式

- 目标是提高信息密度、降低决策成本
- 人只处理三类消息：
  - 需求确认
  - 冲突处理
  - 结果验收
- 治理层负责过滤、提取、沉淀
- 执行层负责真实的多 Agent 工作流执行

关键结论：

- 开放模式可以存在
- 但真正支撑产品价值和付费逻辑的是工作模式

## 8. 工作流与主 Agent 模型

产品心智最终收敛到“负责人模型”：

- 用户前台应该感受到有一个负责人 Agent 对结果负责到底
- 这个负责人 Agent 可以在系统内部调度其他 Agent
- 工作流编排不应该变成用户需要学习的主要概念

这意味着：

- 用户买的是一个能负责到底的负责人
- 工作流只是系统内部语法
- 模式切换和工作流编排可以自动化，但必须可解释、可回退、可审计

## 9. 界面原则

讨论中最强的一条界面原则是：

> 聊天内容可以滚动，确定感不能滚动。

必须始终常驻可见的 3 项信息：

1. 当前负责人
2. 当前阶段
3. 下一次需要人介入的条件

这 3 项构成最小信任仪表盘。

任何会让这 3 项被消息流淹没的设计，都应被视为失败。

## 10. 噪声控制原则

当前群聊里的噪声问题，不应该靠提醒大家少说话来解决。

必须结构化解决：

- 只有高价值消息类型进入主时间线
- 长推理沉淀成 artifact，而不是直接塞进群聊
- 消息要驱动状态变化，而不是成为自由漂浮文本
- 事实、分歧、审批、责任都要被提炼进决策账本
- 主聊天区只显示真正影响人判断的进展、冲突和审批事项

## 11. 产品边界

讨论中也明确区分了 OpenTeams 与相邻产品：

- 普通聊天产品卖的是连接与表达
- 单 Agent 产品卖的是低摩擦执行
- OpenTeams 卖的是复杂问题协同下的对齐与治理

一个场景只有在“错误对齐的代价高于沟通成本”时，OpenTeams 才真正成立。

## 12. 可验证性与审计

整轮讨论都在反对靠想象定义产品。

OpenTeams 必须靠真实工作流观察和对照实验来证明价值。

候选指标包括：

- 有效消息占比
- 人工拍板次数
- 共识形成时长
- 返工率
- 交接错误率
- 上下文丢失率
- 人工 review 成本

并且每一条对外承诺，都必须映射到：

- 一个内部机制
- 一个触发条件
- 一条可审计证据链

## 13. MVP 方向

讨论中最现实的 MVP 方向非常克制：

- 先只做工作模式
- 限制 1 到 3 个可见 Agent
- 先做单层工作流
- 严格限制用户可见消息类型
- 先跑通“需求确认 -> 执行 -> 验收”闭环
- 暂缓把开放模式做成正式产品主面

## 14. 当前产品定义草稿

当前较好的内部表述：

- OpenTeams 是一个以群聊为统一入口、以治理层提纯信息、以多 Agent 工作流交付结果的人机协作系统。
- OpenTeams 帮助用户处理高代价复杂协作问题，而不是被消息噪声淹没。
- OpenTeams 卖的不是热闹讨论，而是可审计的收束、介入和责任机制。

## 15. 仍待验证的问题

还有几个关键问题没有被真正验证：

1. 哪类用户最频繁、最强烈地感受到这种痛点？
2. 第一滩头场景应该选哪一个？
3. 系统何时自动收紧进入工作模式，阈值如何定义？
4. 工作流可以暴露到什么程度而不增加心智负担？
5. 决策账本最小应该长什么样，才能真正带来信任感？

## 16. 最重要的结论

- 不要把 OpenTeams 定义成“多 Agent 群聊工具”。
- 不要让群聊承担主存储层。
- 治理层是本体，群聊是介入表面。
- 用户前台心智必须极简：谁负责、做到哪、何时该我介入。
- MVP 必须围绕工作模式和可验证协作质量展开。
