# OpenTeams 产品计划草案

## 1. 产品目标

把 OpenTeams 做成一个以人为中心的协作系统，让一个人能够协调多个 AI Agent 完成复杂工作，同时不被消息噪声吞没。

第一阶段目标不是做出最强大的多 Agent 平台。

第一阶段目标是证明：在一个高价值工作场景里，OpenTeams 能真实降低协作成本。

## 2. 产品命题

OpenTeams 应该在以下条件下成立：

- 任务不确定性高
- 多种视角同时有价值
- 对齐错误成本高
- 最终仍然需要人承担责任和拍板

如果一个问题本来就适合单 Agent 或普通聊天工具，OpenTeams 不应该强行介入。

## 3. 产品定义

### 用户视角定义

OpenTeams 给用户一个统一协作入口。用户面前看到的是一个对结果负责到底的主 Agent，它会在内部调度其他 Agent、汇总进展、在必要时请求人类介入，并最终交付一个可验收结果。

### 系统视角定义

OpenTeams 由三层组成：

1. 协作入口层：用户可见的聊天与控制界面。
2. 治理层：准入、路由、提取、收束、审批、退出、审计。
3. 执行层：内部多 Agent 工作流。

## 4. 目标用户与滩头场景

### 推荐首批目标用户

- 技术型创业者
- 产品意识较强的工程师
- 带小团队的工程经理

### 推荐第一滩头场景

- 故障处理与复杂排错

选择理由：

- 痛点明显且代价高
- 天然需要多角色视角
- 人类拍板仍不可替代
- 协作失败容易观测
- 指标容易设计

后续可扩展场景：

- 高风险决策评审
- 内容生产与上线协同

## 5. MVP 范围

### In Scope

- 只做工作模式
- 可见 Agent 限制为 1 到 3 个
- 对外只暴露一个主负责人 Agent
- 内部先支持单层工作流
- 用户主时间线只保留三类消息：
  - 需求确认
  - 冲突处理
  - 结果验收
- 提供结构化决策账本
- 提供常驻最小信任仪表盘
- 支持可解释、可回退的自动治理
- 保留完整审计链路

### Out of Scope

- 开放模式作为正式主卖点
- 大规模 Agent 社交群聊
- 面向用户的复杂工作流编排器
- 大规模 Agent swarm
- 无边界自由讨论主时间线

## 6. 核心体验原则

### 原则 1：负责人优先

用户必须始终感觉到有一个 Agent 在对结果负责。

### 原则 2：确定感优先于活跃度

界面优化目标不是消息热闹，而是信任和清晰。

### 原则 3：聊天是表层，不是存储层

重要事实、分歧、审批、责任都应该进入结构化系统，而不是淹没在聊天记录里。

### 原则 4：自动化必须可解释

系统如果自动收紧规则、切换模式、升级冲突或改变执行策略，用户必须知道为什么，并且能回退。

## 7. 核心产品模块

### 7.1 最小信任仪表盘

始终显示三项信息：

- 当前负责人
- 当前阶段
- 下一次需要人介入的条件

验收标准：

- 工作模式下始终可见
- 能随状态自动更新
- 用户无需翻聊天记录也能理解当前系统状态

### 7.2 治理引擎

职责：

- 判断任务是否进入协作模式
- 控制谁该发言、谁不该发言
- 给消息分型并判断价值密度
- 把事实、分歧、审批提炼进决策账本
- 只在必须时把事项升级给人
- 决定何时退出协作转执行或结束验收

验收标准：

- 工作模式下不允许无约束消息流
- 所有被展示的消息都应对应状态变化或决策需要
- 所有审批和责任变化都可追溯

### 7.3 决策账本

结构化记录：

- 当前事实
- 当前分歧
- 候选方案
- 已批准方案
- 当前责任人
- 最终验收结果

验收标准：

- 能解释为什么得出当前结论
- 能清楚展示未解决分歧
- 能脱离聊天记录单独存在并被复用

### 7.4 执行工作流

职责：

- 主 Agent 可在内部拉起子 Agent
- 用简单方式暴露执行进展
- 支持人在关键节点中断或介入
- 最终输出可验收结果包

验收标准：

- 最简单工作流可以只有 1 个 Agent
- 复杂工作流默认隐藏在系统内部
- 所有工作流最终都必须输出可验收结果

## 8. 工作模式消息契约

工作模式主时间线只允许三类消息：

1. 需求确认
2. 冲突升级
3. 结果验收

其余内容应被：

- 折叠
- 转入决策账本
- 保存为 artifact
- 或留在执行日志里

## 9. 核心指标与验证

### 北极星目标

在不削弱责任与可控性的前提下，降低复杂协作成本。

### MVP 指标

- 有效消息占比
- 单任务人工介入次数
- 共识形成时长
- 验收完成时长
- 验收后返工率
- 交接错误数
- 用户是否能仅凭仪表盘理解当前状态

### 需要尽早定义的失败阈值

- 单个议题主时间线消息数过多
- 责任人频繁变化且不可理解
- 用户必须翻聊天记录才能理解当前阶段
- 主 Agent 规划工作流后被人工频繁修正

## 10. 产品叙事草稿

### 对外叙事

OpenTeams 是一个面向复杂工作的协作系统。它不是给你更多聊天，而是给你一个可负责到底的 AI 负责人、被治理过滤的信息流，以及可验收、可审计的结果。

### 对内叙事

OpenTeams 必须在真实使用中证明三件事：

1. 讨论正在收敛
2. 系统知道何时开始失控
3. 正确的人能在正确时机介入

## 11. 推进阶段

### 阶段 0：发现与验证

任务：

- 选定第一滩头场景
- 观察 5 到 10 个真实用户当前如何完成这个工作
- 对比“当前流程 vs OpenTeams 原型”
- 建立基线指标

退出条件：

- 已证明协作成本是真实痛点
- 已证明存在值得付费的量化改进

### 阶段 1：MVP

构建内容：

- 工作模式
- 主负责人 Agent
- 最小信任仪表盘
- 治理引擎 v1
- 决策账本 v1
- 简单执行工作流
- 可验收结果单

退出条件：

- 用户能完成“需求确认 -> 执行 -> 验收”闭环
- 噪声没有失控
- 用户信任仪表盘提供的负责人、阶段和介入条件

### 阶段 2：扩展

候选方向：

- 更丰富的工作流可视化
- 可复用工作流模板
- 面向不同场景的治理策略包
- 作为入口层的开放模式
- 基于历史决策账本的知识复用

## 12. 关键风险

1. 工作流暴露过多，导致心智负担过重。
2. 主 Agent 规划质量不稳定。
3. 自动治理像黑箱，用户不信任。
4. 工作模式仍然退化成噪声群聊。
5. 与单 Agent 工具相比，价值差异不够强。

## 13. 立即下一步

1. 选定第一滩头场景。
2. 把治理层画成具体产品流程图。
3. 定义决策账本的数据结构。
4. 设计常驻不滚动的最小信任仪表盘。
5. 做一个严格执行三类消息契约的工作模式原型。
6. 设计第一轮“当前流程 vs OpenTeams”的对照验证方案。
